## 1. Общая картина за 30 секунд

1. Бот **слушает блокчейн** (новые монеты и сделки)
2. На каждый новый токен смотрит **историю разработчика (dev)** и **живую ленту сделок** ~5 секунд.
3. Считает **баллы (score)** → получает **tier**: Skip / **A** / **A+**.
4. Если прошёл все ворота — **покупает** (размер зависит от tier и лимитов).
5. Пока позиция открыта — каждые ~100 мс пересчитывает **PnL по кривой**, **живой score**, **фазу рынка** и решает: держать, частично продать (TP), резать (SL), выйти по таймеру (time-kill) или по «угасанию пампа» (momentum decay).
6. После **полного** закрытия — в фоне **30 минут** снимает mcap (пока кривая жива или по рыночной цене после миграции) и пишет в БД: «улетела ли дальше или мы вышли рано».

**Главная идея:** вход — это «качество dev + живой памп в окне», выход — это «не один триггер», а набор правил с приоритетом и профилем сделки (weak / strong / runner).

---

## 2. Как бот видит новый токен

- В **реальном времени** приходят события pump.fun создание монеты и каждая сделка на кривой.
- Событие **Create** — монета создана; бот заводит для неё **«ведро» (bucket)** — виртуальный пул с кривой bonding curve и списком покупателей/продавцов.
- Каждая **сделка (Trade)** обновляет цену на кривой, объём, кто купил (обычный покупатель, снайпер, сам dev).

Если dev не прошёл ранние фильтры или score = Skip — дальше по этому mint бот **не покупает** (mint попадает в «закрытые» и не переоткрывается).

## 3. Фильтр разработчика (creator / dev)

**До** подсчёта score бот смотрит в базу: **какие монеты этот dev уже запускал** и как они жили.

Параметры в `filter_config.yaml` → блок `**creator_config`** (пороги «история dev нормальная»):


| Параметр                              | Смысл простыми словами                                               |
| ------------------------------------- | -------------------------------------------------------------------- |
| `min_total_coins` / `max_total_coins` | Сколько прошлых монет у dev — не слишком мало опыта, не спам сотнями |
| `min_median_market_cap`               | Медианный пик mcap прошлых монет dev (в SOL)                         |
| `min_trader_pnl_average`              | Средний PnL трейдеров на его прошлых монетах                         |
| `min_total_holders_average`           | Среднее число уникальных покупателей                                 |
| `min_average_volume`                  | Средний объём торгов                                                 |
| `min_median_total_trades`             | Медиана числа сделок на монету                                       |


**Если dev не прошёл** — в логе: `[FILTER] … rejected by creator_config` или `no creator history`. Score **не считается** — экономим время и не входим в мусорных dev.

Дополнительно со временем копится **dev ranker** (файл состояния): dev помечается как сильный/слабый по исходам прошлых сделок бота — это даёт баллы в score (`dev_history_strong`, `dev_ranker_*`).

## 4. Окно scoring (~5.5 сек) и V3 tape

После прохождения creator-фильтра бот **не покупает мгновенно**. Он **наблюдает** монету:

- `**scoring_window_ms`** (обычно 5500 мс) — сколько миллисекунд смотрим ленту.
- Внутри окна несколько **срезов** (`buyer_velocity_slices`, обычно 3) — как меняются покупатели и давление продаж.

### Что такое V3 tape

**V3 tape** — это сжатая «медицинская карта» первых секунд, которая потом видна в dashboard и пишется в БД при входе:


| Поле в интерфейсе               | Что значит                                           |
| ------------------------------- | ---------------------------------------------------- |
| **bv** (buyer velocity persist) | Покупатели приходят **устойчиво** или рывками        |
| **sp** (sell pressure)          | Насколько агрессивны продажи vs покупки              |
| **ab** (absorb)                 | Покупатели **вбирают** продажи (хороший знак)        |
| **d0** (dumps)                  | Сколько «порций» похожих продаж подряд (плохой знак) |
| **sm0** (smart exits)           | Сколько «умных» кошельков уже вышли в ноль           |


Эти же числа влияют на баллы: `buyer_velocity_persistent`, `sell_pressure_high`, `absorb_strong`, `repeat_dump_penalty`, `smart_early_exit_dump`.

Окно может **закончиться раньше**, если mcap за время окна вырос достаточно сильно (быстрый памп) — в логе `early_exit=true`.

## 5. Score, tier A / A+ и momentum_good

### Как считается score

Бот складывает **баллы и штрафы** из списка факторов (`scoring.weights` в конфиге). Примеры:

- **Плюс:** сильная история dev, много покупателей, хороший buy/sell, объём, absorb, momentum_good, умные кошельки.
- **Минус:** мало покупателей, bundle (подозрительно одинаковые покупки), перегрев momentum, давление продаж, повторные дампы.

Итог — число `**score`**. По порогам:


| Tier     | Обычно нужно score ≥     |
| -------- | ------------------------ |
| **A+**   | `a_plus_threshold` (9)   |
| **A**    | `a_threshold` (6)        |
| **Skip** | ниже A — **не покупаем** |


Размер ставки: **A+** → `scoring.size.a_plus_sol` (0.4 SOL), **A** → `a_sol` (0.3 SOL), но не больше операторского лимита `buy_config.amount_sol`.

### momentum_good — отдельно важно

**momentum_good** — не просто «цена выросла», а **рост mcap внутри scoring-окна** от первого среза к **пику** в допустимом коридоре:

- от `**momentum_good_low_pct`** (сейчас 3%) 
- до `**momentum_good_high_pct**` (30%)

Если рост слишком слабый — в items не будет `momentum_good`, и при `**require_momentum_good: true**` (live) бот **не купит**, даже если tier = A.

Если рост **слишком бешеный** (выше `momentum_overheated_pct`, ~55%) — штраф `momentum_overheated`: «уже поздно, перегрев».

**Почему так:** tier A без живого пампа в окне часто = вход в стоячий или слабый токен → много ложных SL.

## 6. Когда принимается решение BUY

Цепочка **всех ворот** (все должны пройти):

1. Токен создан, лента идёт.
2. **creator_config** — dev ок.
3. **Scoring window** — собрали tape и features.
4. **tier ≠ Skip**.
5. **Live-гейты** (если `execution.mode: live`):
  - `require_momentum_good` — в breakdown есть `momentum_good`;
  - `minimum_tier_for_buy` — обычно **A** (допускаются и A, и A+); можно поставить **APlus** — только топ.
6. **Strategy controller** — не превышены лимиты дня:
  - `max_open_positions` — сколько позиций одновременно;
  - `max_trades_per_day`;
  - `max_daily_loss_sol` / `daily_profit_lock_sol`;
  - пауза после серии лоссов (`loss_streak_`*);
  - «плохой рынок» (`market_regime_*`) — если последние N сделок в минусе.
7. Бот **не на паузе** (оператор не нажал pause в dashboard).
8. Mint ещё **не был полностью закрыт** ранее.

В логе успешного входа:

```text
[BUY GATE] … tier=APlus tier_sol=0.400 … final_amount_sol=0.400
[BUY] Opened … | mcap=… SOL | spent=0.4000 SOL
```

### Размер покупки

```text
final_amount_sol = min(tier_sol, operator_cap)
```

- **tier_sol** — 0.4 для A+, 0.3 для A.
- **operator_cap** — `buy_config.amount_sol` (ползунок размера ставки в dashboard).

После сигнала покупки есть **небольшая задержка (~800 ms)** перед отправкой транзакции — чтобы лента успела стабилизироваться.

### Сразу после покупки (защита)

Если **фактический mcap после fill** сильно выше, чем при score (`fill_mcap_abort_max_ratio`, напр. 2×) — бот **сразу продаёт всё** (spike abort): «опоздали, цена уехала до нашего входа».

## 7. Пока позиция открыта: что бот «видит»

Каждые ~100 ms для открытой позиции:

- **Текущий mcap** с bonding curve (с **медианным фильтром** — отсекаются одиночные «битые» тики).
- **PnL %** относительно mcap входа.
- **Live score** — упрощённая оценка «живости» ленты (обновляется каждые ~3 сек).
- **Фаза V4:** Exploration → Momentum → Expansion → **Distribution** (распределение / затухание).
- **Профиль выхода:** weak / neutral / **strong** / **runner** (может повыситься в runtime до runner).

## 8. Как работает продажа (приоритет правил)

Бот проверяет условия **сверху вниз**. Первое сработавшее — причина в **TX LOG** / **SELL**.

### 8.1 Momentum decay (V4)

**Смысл:** памп **закончился** — падает скорость покупок, растёт sell pressure, или фаза **Distribution** и live score упал до низа.

- Первые `**exit_grace_secs`** (3 сек) после входа decay **не режет** — чтобы не выйти на микро-шуме сразу после buy.
- Выход: `MOMENTUM DECAY` или `MOMENTUM DECAY [distribution]`.

### 8.2 Time kill

**Смысл:** «Дали время выстрелить — не выстрелил».

- Если через **N секунд** profit **ниже** `time_kill_min_profit_pct` (10%) — закрываем всё.
- **N адаптивный** (`time_kill_adaptive: true`):
  - слабый старт → ~22 сек (`time_kill_weak_secs`);
  - нейтральный → ~32 сек;
  - сильный → ~60 сек (`time_kill_strong_secs`);
  - смотрит на buyers/b2s при входе и скорость mcap в первые секунды.
- Если уже сняли **TP1/TP2** и profit ≥ 8% — time-kill для strong/runner **может быть пропущен**.

### 8.3 Stop-loss (SL)

**Смысл:** защита от просадки по **mcap**, не по «одному глючному тику».

- Стартовый пол: `**exit_profit_floor`** (−16% по mcap).
- Первые `**sl_grace_secs**` (5 сек) SL **не активен**.
- SL срабатывает только если **3 тика подряд** (sl_confirm_ticks) mcap ≤ пола.
- После сильного профита включается **profit lock (лестница)** — пол поднимается (например после +50% пика → пол +15%): выход `SL (floor 15.0%, 3 ticks, …)`.

### 8.4 Take profit (TP) — частичные продажи

Не «всё сразу», а **лестница**:


| Уровень | Пример runner | Что делает                         |
| ------- | ------------- | ---------------------------------- |
| TP1     | +50% mcap     | Продаёт **10%** оставшихся токенов |
| TP2     | +80%          | Ещё 10%                            |
| TP3+    | +150%, +300%… | Дальше по конфигу профиля          |


В логе: `TP1 (50%) [runner|momentum] | sold=10% of holdings`.

Профили **weak / neutral / strong / runner** имеют **разные** пороги TP и % продажи — runner продаёт **малыми долями** и дольше держит остаток под большой трейлинг.

### 8.5 Trailing stop (трейлинг)

**Смысл:** зафиксировать часть пампа, но дать «дохнуть» от пика.

- Запоминается **пик mcap** в позиции.
- Если цена упала на **X% от пика** (зависит от профиля и live score, runner ~38%) — полный выход: `TRAILING EXIT (38% trail)`.

### 8.6 Потолок mcap

Если mcap ≥ `mcap_ceiling_sol` (по умолчанию ~350 SOL) — частичная фиксация (`MCAP CEILING`, ~65% sell), остаток moonbag под trailing/Jupiter.

## 9. Профили выхода: weak, neutral, strong, runner

При открытии позиции бот назначает **профиль** (по силе входа: buyers, b2s, smart, bundle…):


| Профиль     | Характер                 | Типичное поведение                                              |
| ----------- | ------------------------ | --------------------------------------------------------------- |
| **weak**    | Слабый старт             | Быстрее TP, жёстче трейлинг, короче терпение                    |
| **neutral** | Середина                 | Умеренные TP/трейлинг                                           |
| **strong**  | Хорошие buyers/volume    | TP дальше, больше шанс дождаться                                |
| **runner**  | Очень сильный setup / A+ | Маленькие частичные TP, широкий трейлинг, цель — «долгий забег» |


В runtime при сильной ленте позиция может **повыситься до runner** (`runtime_runner_upgrade_enabled`).

**HOLD mode** (редко) — особый режим «не трогать пока лента идеальная» (очень высокий live score, smart wallets, нет bundle).

## 10. Post-exit tracking — что было после нашего SELL

После **100% закрытия** (не после частичного TP) бот **30 минут** в фоне снимает mcap:

**Расписание (секунды после выхода):**  
`10, 30, 50, 70, 100, 180, 240, 300` → затем `600 (10m), 900 (15m), 1800 (30m)`.

**Откуда берётся цена:**

1. С **кривой pump** — пока монета ещё на bonding curve;
2. С **внешнего рынка** — если кривая уже завершена и монета торгуется на DEX (оценка по актуальной цене свапа).

**В БД сохраняется:**

- mcap на каждой отметке + **% от exit_mcap**;
- **max / min** mcap за окно;
- `**post_exit_time_to_max_secs`** — через сколько секунд был пик после выхода;
- `**post_exit_time_to_min_secs**` — через сколько секунд было дно.

**Зачем:** понять «мы вышли рано» (после SELL +60% через 70 сек) или «правильно резали» (после SELL только −10% и падение).

В логе бота видно время среза, mcap и откуда взята цена (кривая или рынок после миграции), а в конце — max/min и через сколько секунд был пик или дно.

## 12. Dashboard — как читать экран

Панель **подключается к работающему боту** и показывает то же, что происходит в торговле: открытые позиции, сделки, историю.

### OPEN (открытые позиции)

- Монеты, в которых **сейчас** лежат деньги.
- PnL в реальном времени, mcap, сколько ещё держим.
- Статусы: floor (текущий SL-пол), [TP1], [TRAIL], [CLOSING]…

### TX LOG (лог транзакций)

Хронология событий:


| Тип             | Значение                                                                                                |
| --------------- | ------------------------------------------------------------------------------------------------------- |
| **BUY**         | Покупка (live или demo)                                                                                 |
| **SELL**        | Продажа — смотри **reason** (TP1, SL, TIME KILL, MOMENTUM DECAY, TRAILING…)                             |
| Колонка **PnL** | `mcap +X%` и `SOL +Y%` — **mcap** = движение по кривой, **SOL** = факт кошелька (важно при расхождении) |


**Почему mcap и SOL могут расходиться:** проскальзывание, комиссии, cashback на curve, частичные продажи, разная цена после ухода монеты с кривой на DEX.

### HISTORY (история)

- Прошлые **полные** закрытия из БД.
- PnL, причина, tier/score из meta.
- Клик → **график** сделки (кривая mcap, маркеры входа/выхода).
- После закрытия список **обновляется** автоматически.

### Прочее в UI

- **Balance** — сколько SOL на торговом кошельке (в live-режиме).
- **Buy size** — операторский потолок ставки (не выше tier).
- **Pause** — новые покупки стоп, открытые позиции **дальше управляются**.

## 13. filter_config.yaml — шпаргалка по блокам

Файл `**filter_config.yaml`** — главные «ручки» торговой логики (часть изменений вступает в силу после перезапуска бота).

### `creator_config`

Пороги истории dev **до** score (см. раздел 3).

### `buy_config`


| Параметр                     | За что отвечает                                             |
| ---------------------------- | ----------------------------------------------------------- |
| `amount_sol`                 | Макс. ставка с панели (оператор)                            |
| `tp1_pct` / `tp1_sell_pct`   | Первый take-profit: при +X% mcap продать Y% **остатка**     |
| `exit_profit_floor`          | Стартовый SL по mcap (напр. −16%)                           |
| `sl_grace_secs`              | Секунд после входа без SL                                   |
| `sl_confirm_ticks`           | Сколько тиков подряд ниже пола для SL                       |
| `exit_mcap_median_ticks`     | Сглаживание mcap (анти-глюк)                                |
| `time_kill_*`                | Таймер «не выстрелил — выходим»                             |
| `trailing_stop_drawdown_pct` | Базовый трейлинг (если V4 выкл.)                            |
| `fill_mcap_abort_*`          | Экстренный выход если fill слишком дорогой                  |
| `**exit_v4`**                | Вся умная логика фаз, профилей, decay, лестницы profit-lock |


#### Внутри `exit_v4` (важное)


| Параметр                                 | За что                                            |
| ---------------------------------------- | ------------------------------------------------- |
| `exit_grace_secs`                        | Пауза после buy без decay-exit                    |
| `phase_distribution_max_score`           | При каком live score считаем фазу «распределение» |
| `momentum_decay_exit_enabled`            | Включить выход по затуханию                       |
| `profit_staircase_enabled`               | Поднимать SL-пол после большого профита           |
| `runtime_runner_upgrade_enabled`         | Может повысить до runner по ленте                 |
| `weak` / `neutral` / `strong` / `runner` | Свои TP% и trailing для каждого типа              |


### `scoring`


| Параметр                           | За что                                   |
| ---------------------------------- | ---------------------------------------- |
| `scoring_window_ms`                | Длина окна перед buy                     |
| `a_threshold` / `a_plus_threshold` | Пороги tier A и A+                       |
| `require_momentum_good`            | Live: без роста mcap в окне — не buy     |
| `minimum_tier_for_buy`             | Минимум A или только A+                  |
| `weights` / `thresholds`           | Баллы за факторы и пороги «хорошо/плохо» |
| `size.a_sol` / `a_plus_sol`        | Размер ставки по tier                    |


### `strategy`

Дневные лимиты и паузы (**не трогает** уже открытые позиции).

### `execution`

| `mode: demo` | Учебный режим: сделки симулируются, баланс виртуальный |
| `mode: live` | Боевой режим: реальные покупки и продажи на кошельке |
| `live.slippage_bps` | Допуск проскальзывания при исполнении |

### `persistence` / `learning`

Накопленная память по dev и «умным» кошелькам; блок learning со временем слегка подкручивает пороги score по накопленным сделкам (осторожно, с лимитами).

## 14. Типичные причины в SELL — расшифровка


| Reason в логе                   | Простыми словами                                |
| ------------------------------- | ----------------------------------------------- |
| `TP1 (50%) [runner|…]`          | Первая фиксация прибыли, часть позиции          |
| `TP2 …` / `TP3 …`               | Следующие ступени                               |
| `SL (floor -16.0%, 3 ticks, …)` | Стоп по mcap, подтверждён 3 тиками              |
| `SL (floor 15.0%, …)`           | Profit-lock: защита прибыли после большого пика |
| `TIME KILL`                     | Время вышло, профит так и не стал ≥ 10%         |
| `MOMENTUM DECAY`                | Лента умерла (скорость/продажи)                 |
| `MOMENTUM DECAY [distribution]` | Фаза «распределение», score на дне              |
| `TRAILING EXIT (38% trail)`     | Откат от пика на 38%                            |
| `MCAP CEILING`                  | Высокий mcap: partial lock + moonbag (trailing) |
| fill spike abort                | Вход слишком поздний vs score                   |

## 15. Режимы demo и live

- **Demo** — «бумажная» торговля для проверки логики; баланс из `start_balance_sol`; гейты momentum/tier могут быть мягче.
- **Live** — реальные деньги на кошельке; строже `require_momentum_good` и tier.

## 16. Если нужно «почему именно эта сделка»

1. **Лог бота** — строки `[SCORE]`, `[BUY GATE]`, `[EXIT V4]`, `[SELL]`, `[POST-EXIT]`.
2. **Dashboard TX LOG** — хронология BUY/SELL с reason и PnL.
3. **HISTORY + график** — визуально entry/exit на кривой.
4. **История в БД** — post-exit % и time_to_max/min после заполнения (~30 мин).
